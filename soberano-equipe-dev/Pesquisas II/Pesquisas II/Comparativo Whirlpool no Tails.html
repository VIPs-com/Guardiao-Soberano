Comparativo: Whirlpool no Tails vs Whirlpool no Whonix

Perfeito. Vou fazer o comparativo completo entre Tails vs Whonix para rodar Whirlpool e, em seguida, detalhar o passo a passo da Opção B: migrar para Whonix Workstation persistente, incluindo a integração com Coldcard, Electrum Personal Server e eigenwallet.

Comparativo: Whirlpool no Tails vs Whirlpool no Whonix

Tabela comparativa direta

Análise qualitativa

Tails: o cofre portátil que você abre só quando precisa

Vantagens reais:

  • Não deixa rastro no hardware. Se você mora em país hostil a cripto, o Tails não deixa vestígio no disco do computador.
  • O sistema inteiro é imutável. Um malware que infecte o Sparrow morre no reboot (a não ser que grave no Persistent Storage, o que é difícil).
  • Portabilidade extrema: cabe num pendrive no bolso. Você pode mixar em qualquer laptop emprestado.
  • Custo zero em infraestrutura: não precisa de hypervisor, nem de VM, nem de node dedicado.

Desvantagens reais:

  • A persistência é uma gambiarra elegante, mas gambiarra. O Tails não foi projetado para manter estado de aplicações complexas como o Whirlpool.
  • Cada sessão de mixagem longa é uma oração silenciosa para a energia não cair e o USB não pifar.
  • Restaurar o estado do Whirlpool após perda do Persistent Storage é chato e demorado, embora seguro.
  • Se você mixa com frequência (vários UTXOs, vários pools), o ciclo de boot → carregar carteira → iniciar Whirlpool → esperar rounds → desligar fica desgastante.

Conclusão para Tails:
Use Tails se você faz coinjoins táticos — uma vez por mês, prepara UTXOs para swaps futuros, não tem pressa. É o cofre que você vai buscar no banco quando precisa, não o cofre que fica aberto no quarto.

Whonix: a fortaleza que fica de prontidão

Vantagens reais:

  • A arquitetura de duas VMs é uma obra-prima: a Workstation não tem IP real, não conhece sua localização geográfica, não sabe seu endereço MAC. Mesmo que o Sparrow seja 0-day explorado, o atacante está preso numa VM sem rota para a internet — ele só vê o Gateway, que só fala Tor.
  • Snapshots são um superpoder. Antes de cada round de Whirlpool, tire snapshot. Se algo corromper, volta em 30 segundos.
  • Pode mixar 24/7. Deixa a VM rodando enquanto trabalha, dorme, viaja. O Whirlpool adora tempo de uptime — mais rounds, mais pares, anonset sobe mais rápido.
  • Integração com node próprio (EPS) é natural e elegante. Você pode ter uma terceira VM só com Bitcoin Core + EPS, toda comunicação interna criptografada e via Tor.
  • Backup é copiar arquivo. Sem ritual de clonagem de USB, sem dd nervoso.

Desvantagens reais:

  • Complexidade de setup não é trivial. Instalar Whonix, configurar rede interna, pastas compartilhadas, USB passthrough — leva uma tarde inteira na primeira vez.
  • Exige hardware parrudo. Com host Linux + Gateway VM + Workstation VM + (opcional) Node VM, 16 GB de RAM é o mínimo confortável.
  • O host não é anônimo. Seu sistema operacional base (Windows, Linux, macOS) sabe que você está rodando Whonix. Em país opressor, isso pode ser um problema. O Tails não deixa rastro; o Whonix deixa a VM no seu disco.
  • Malware que escape da VM para o host é cenário de ataque avançado (ex: hipervisor escape), mas não impossível. No Tails, não há host para escapar — é o hardware direto.

Conclusão para Whonix:
Use Whonix se você é operador frequente — mantém node próprio, faz coinjoins semanais, swaps regulares, e quer a conveniência de um sistema sempre pronto, com backup fácil e resiliência contra falhas. É a fortaleza que fica de prontidão no terreno, não o cofre que você vai buscar.

Matriz de decisão rápida

Estratégia híbrida (a que eu recomendaria para você)

Com base no seu protocolo completo (Tails para swaps, Whirlpool, eigenwallet, Feather), você pode ter os dois, em funções separadas:

Fluxo:

  1. No Whonix, você mixa BTC continuamente até ter outputs pós-coinjoin prontos.
  2. Move os outputs limpos para uma carteira Postmix no Whonix.
  3. Quando vai fazer um swap, transfere só o necessário do Whonix para um endereço da carteira no Tails.
  4. No Tails, executa o swap (eigenwallet), recebe XMR no Feather.
  5. O Whonix nunca toca em Monero; o Tails nunca guarda estado de longo prazo. Segregação de funções = segurança em camadas.

Isso resolve o dilema: Whonix para o trabalho pesado e persistente (coinjoin), Tails para a operação cirúrgica e descartável (swap).

Opção B: Guia de migração para Whonix Workstation Persistente

Se você decidir pelo Whonix (seja como ambiente principal ou como complemento ao Tails), aqui está o passo a passo completo.

Pré-requisitos de hardware e software

  • Host: Linux (Debian/Ubuntu recomendado), Windows ou macOS
  • RAM: 16 GB mínimo (8 GB host + 2 GB Gateway + 6 GB Workstation)
  • Armazenamento: 100 GB livres para as VMs + blockchain do Bitcoin Core (se rodar node)
  • VirtualBox 7.x ou KVM/QEMU instalado
  • Coldcard Mk5 com cabo USB ou adaptador MicroSD
  • Paciência para 2-3 horas de configuração inicial

Passo 1 — Baixar e verificar Whonix

1.1 Obter as imagens oficiais

No host (não no Tails), via Tor Browser ou torify wget:

# Baixar Whonix para VirtualBox (formato .ova)
torify wget https://download.whonix.org/ova/Whonix-XFCE-*.ova

# Baixar assinatura e chave
torify wget https://download.whonix.org/ova/Whonix-XFCE-*.ova.asc
gpg --keyserver hkps://keys.openpgp.org --recv-keys 916B8D99C38EAF5E8ADC7A2A8D66066A2EEACCDA
gpg --verify Whonix-XFCE-*.ova.asc Whonix-XFCE-*.ova

A chave de assinatura do Whonix (2EEACCDA) deve ser verificada cruzadamente no site oficial whonix.org e no repositório GitHub.

Passo 2 — Importar e configurar as VMs no VirtualBox

2.1 Importar o .ova

  • VirtualBox → File → Import Appliance
  • Selecionar o arquivo .ova
  • Importar com configurações padrão (2 VMs serão criadas: Whonix-Gateway e Whonix-Workstation)

2.2 Configurar rede interna

As VMs já vêm com rede interna configurada (Whonix), mas verifique:

Whonix-Gateway:

  • Settings → Network → Adapter 1: NAT (para sair para internet)
  • Adapter 2: Internal Network, nome Whonix

Whonix-Workstation:

  • Settings → Network → Adapter 1: Internal Network, nome Whonix
  • NENHUM outro adaptador. A Workstation só enxerga o Gateway.

2.3 Aumentar recursos da Workstation (para Whirlpool pesado)

  • Settings → System → RAM: 4096 MB (ou 6144 MB se tiver folga)
  • Settings → System → Processor: 2-4 CPUs
  • Settings → Display → Video Memory: 128 MB
  • Settings → Storage → disco virtual: 50 GB (pré-alocado se possível)

2.4 Configurar pastas compartilhadas (para PSBT com Coldcard)

Isso é crítico para seu fluxo PSBT. Você precisa passar arquivos entre o host (onde está o Coldcard) e a Workstation (onde está o Sparrow).

Opção A: Pasta compartilhada do VirtualBox

  • Settings → Shared Folders → Add
  • Folder Path: /home/seuuser/psbt_bridge (crie essa pasta no host)
  • Folder Name: psbt_bridge
  • Read-only: Não
  • Auto-mount: Sim
  • Mount point: /mnt/psbt_bridge

Opção B: USB passthrough (para Coldcard direto)

  • Settings → USB → Enable USB Controller → USB 2.0 ou 3.0
  • Iniciar a VM, depois Devices → USB → selecionar Coldcard (aparece como dispositivo USB)

A Opção A é mais estável e recomendada. Você copia o PSBT do Coldcard para a pasta psbt_bridge no host, e a Workstation acessa via /mnt/psbt_bridge.

Passo 3 — Primeiro boot e configuração inicial

3.1 Iniciar o Gateway primeiro

SEMPRE inicie o Whonix-Gateway antes da Workstation. Aguarde o Tor estar pronto (ícone verde no systray).

3.2 Iniciar a Workstation

Faça login: usuário user, senha changeme (troque imediatamente).

passwd
# Digite a senha atual (changeme) e depois a nova (20+ caracteres, anote no KeePassXC)

3.3 Atualizar o sistema

sudo apt update && sudo apt full-upgrade -y
# Isso pode levar 15-30 min na primeira vez, pois tudo passa pelo Tor

3.4 Verificar que o Tor está funcionando

curl --socks5-hostname 127.0.0.1:9050 https://check.torproject.org
# Deve retornar "Congratulations. This browser is configured to use Tor."

Passo 4 — Instalar ferramentas cripto na Workstation

4.1 Sparrow Wallet

# Baixar via Tor (já forçado pelo Whonix)
wget https://sparrowwallet.com/downloads/sparrow-*.AppImage
wget https://sparrowwallet.com/downloads/sparrow-*.AppImage.asc

# Verificar PGP (importar chave do Craig Raw primeiro)
gpg --verify sparrow-*.AppImage.asc sparrow-*.AppImage

chmod +x sparrow-*.AppImage
mkdir -p ~/Applications
mv sparrow-*.AppImage ~/Applications/

4.2 KeePassXC (já instalado ou via apt)

sudo apt install keepassxc -y

Crie o banco de dados cifrado e armazene-o em ~/Documents/ (será backupeado com a VM).

4.3 Feather Wallet (para XMR)

wget https://featherwallet.org/files/releases/linux/feather-*.AppImage
# Verificar PGP conforme seu protocolo
chmod +x feather-*.AppImage
mv feather-*.AppImage ~/Applications/

4.4 eigenwallet (para swaps BTC↔XMR)

wget https://eigenwallet.org/releases/eigenwallet-*.AppImage
# Verificar PGP do binarybaron
chmod +x eigenwallet-*.AppImage
mv eigenwallet-*.AppImage ~/Applications/

4.5 Electrum (opcional, já que Sparrow é o foco)

sudo apt install electrum -y
# Ou baixar AppImage do site oficial (recomendado para versão mais recente)

Passo 5 — Configurar Electrum Personal Server (EPS) — Opcional mas recomendado

Se você tem um node Bitcoin rodando (pode ser no host, em outra VM, ou num Raspberry Pi), configure o EPS para que o Sparrow na Workstation se conecte a ele via .onion.

5.1 No node Bitcoin (ex: Raspi ou VM separada)

# Instalar EPS
git clone https://github.com/chris-belcher/electrum-personal-server.git
cd electrum-personal-server

# Configurar para Tor
nano config.ini
# Adicionar:
# [electrum-server]
# tor_host = 127.0.0.1
# tor_port = 9050
# tor_hidden_service = 1

# Rodar EPS
python3 electrum-personal-server.py config.ini

O EPS gerará um endereço .onion. Anote-o.

5.2 No Sparrow da Workstation

  • Preferences → Server → Private Electrum
  • URL: seudominio.onion:50002
  • Use Tor: Marcado
  • Use SSL: Configurar se seu EPS tiver certificado

Passo 6 — Importar carteira watching-only e configurar Whirlpool

Siga o mesmo ritual que você faria no Tails:

6.1 Copiar xpub do Coldcard para a Workstation

  • Coldcard → Advanced → MicroSD → Export Wallet → Sparrow Wallet
  • Copiar o arquivo .json para a pasta compartilhada psbt_bridge(no host)
  • Na Workstation, acessar /mnt/psbt_bridge/ e copiar o arquivo para ~/Documents/

6.2 Criar carteira no Sparrow

  • File → New Wallet → Whirlpool_Whonix
  • Import File → selecionar o .json

6.3 Configurar Whirlpool

  • Aba Whirlpool → Settings:
    • Coordinator: coordinator.sparrowwallet.com.onion
    • Use Tor: Marcado
  • Criar transação Premix (mesmo processo do Tails)
  • Assinar PSBT: Coldcard assina → salva na pasta psbt_bridge → Workstation carrega e transmite

Passo 7 — Backup e snapshots (a grande vantagem do Whonix)

7.1 Snapshot antes de cada etapa crítica

No VirtualBox:

  • Selecionar Whonix-Workstation → Snapshots → Take
  • Nome: pre_whirlpool_setup, premix_confirmed, round_3_complete, etc.

Se algo der errado (ex: coordinator falhar, VM corromper), você restaura o snapshot em segundos.

7.2 Backup completo da VM

Copie a pasta inteira da VM (~/VirtualBox VMs/Whonix-Workstation/) para um HD externo criptografado.

Frequência: após cada sessão de swap ou após completar rounds de Whirlpool.

7.3 Backup do KeePassXC

O banco ~/Documents/whonix_crypto.kdbx deve ser copiado para o host via pasta compartilhada e para backup externo.

Passo 8 — Automação e conveniência (opcional)

8.1 Script de inicialização rápida

Crie ~/start_crypto.sh:

#!/bin/bash
# Iniciar Sparrow
~/Applications/sparrow-*.AppImage &

# Aguardar Sparrow abrir
sleep 10

# Iniciar KeePassXC
keepassxc ~/Documents/whonix_crypto.kdbx &

echo "Ambiente cripto iniciado. Verifique Tor antes de operar."

8.2 Agendamento de backups

No host (não na VM), crie um cron job para copiar a VM a cada semana:

# crontab -e
0 3 * * 0 rsync -av ~/VirtualBox\ VMs/Whonix-Workstation/ /mnt/backup_hd/whonix_backup/

Resumo: Tails vs Whonix para seu caso específico

Minha recomendação final para você

Com base no seu protocolo completo e perfil de desenvolvedor:

Ambiente primário: Whonix Workstation

  • Whirlpool contínuo
  • Sparrow watching-only
  • KeePassXC com metadados

Ambiente cirúrgico: Tails (pendrive separado)

  • Swaps eigenwallet
  • Feather Wallet
  • RoboSats

Ciclo operacional:

  1. Whonix mixa BTC → outputs prontos
  2. Transfere para Tails (transação PSBT assinada no Coldcard)
  3. Tails executa swap → XMR no Feather
  4. Tails desliga, sem rastro
  5. Whonix continua mixando o próximo lote

Assim você tem o melhor dos dois mundos: persistência e conveniência no Whonix, descartabilidade e anonimato máximo no Tails.
