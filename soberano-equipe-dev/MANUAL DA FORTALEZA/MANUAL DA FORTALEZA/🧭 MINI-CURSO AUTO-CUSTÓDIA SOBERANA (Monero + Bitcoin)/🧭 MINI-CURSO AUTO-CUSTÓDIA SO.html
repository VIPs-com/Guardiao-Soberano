Módulo 1 – Fundamentos: O que você realmente possui?

Aula 1.1 – A ilusão do banco e a verdade da seed

Problema: Você acha que seu dinheiro está no computador. Não está.

Analogia: A seed é o tesouro enterrado. O aplicativo é o carro que você usa para chegar até ele. O carro quebra, você pega outro – o tesouro continua lá.

Demonstração no terminal (Feather Wallet): 

# Baixar a Feather (se ainda não tiver)
wget https://featherwallet.org/files/releases/feather-2.6.4.AppImage
chmod +x feather-*.AppImage

# Executar e criar uma carteira (anote a seed em papel – use uma seed falsa para teste)
./feather-*.AppImage

Crie uma carteira com nome teste. Anote as 16 palavras (Polyseed) num papel.
Feche a Feather. Delete o arquivo da carteira:

rm ~/.feather/wallets/teste.keys

Agora reabra a Feather e escolha Restore wallet from seed. Digite as 16 palavras.
O saldo aparece novamente.

Modelo de ameaça: Protege contra perda do computador ou corrupção do arquivo da carteira.
Não protege contra: seed exposta ou senha fraca.

Desafio: Crie uma carteira, apague o arquivo, restaure só com a seed. Comente: “Consegui restaurar!”

Aula 1.2 – Bitcoin vs Monero – duas filosofias

Problema: Bitcoin é público; Monero é privado.

Analogia: Bitcoin = caminhão de vidro transparente (todo mundo vê a carga). Monero = caixa preta lacrada.

Demonstração (não requer terminal, apenas navegador):

  • Acesse mempool.space e cole um endereço Bitcoin público. Você vê saldo e todas as transações.
  • Acesse xmrchain.net e procure o hash de uma transação Monero. Você só vê data e tamanho; nem o valor, nem os endereços.

Terminal – mostrando como o Monero não vaza informação (exemplo com monero-wallet-cli depois de sincronizar): 

# Criar uma carteira Monero de teste
monero-wallet-cli --generate-new-wallet carteira_teste
# O comando acima pede senha e mostra a seed (anote)

Depois, ao executar show_transfers, você vê apenas as suas transações – ninguém mais pode ver.

Modelo de ameaça: Monero protege contra vigilância financeira e rastreamento por terceiros.
Não protege contra: KYC na entrada (a compra na exchange já está ligada ao seu CPF).

Desafio: Acesse um block explorer do Bitcoin e outro do Monero. Perceba a diferença.

Aula 1.3 – Modelagem de ameaça para pessoas comuns

Problema: Você não sabe contra o quê está se protegendo.
Analogia: Não adianta colocar um cadeado se o problema é o telhado furado.

Três ameaças reais e suas proteções:

  1. Pen drive roubado → criptografia LUKS (senha forte).
  2. Coerção física (te obrigam a dar a senha) → VeraCrypt com volume oculto (negação plausível).
  3. Pen drive morre (falha de hardware) → clone com rsync + pen drive B.

O que a tecnologia não resolve (erros humanos comuns):

  • Senha fraca (ex: 123456, data de nascimento).
  • Foto da seed no celular (vai para a nuvem).
  • Contar para pessoas erradas que você tem cripto.
  • Não testar o backup.

Material de apoio no terminal – gerador de senha forte com openssl : 

# Gera uma senha de 20 caracteres aleatórios (use como base para sua senha LUKS)
openssl rand -base64 20

Desafio: Identifique qual das três ameaças é mais real para você. Anote num papel qual proteção vai implementar primeiro.

Desafio: Crie uma carteira, anote seed e restore height. Feche, delete os arquivos, restaure. O saldo deve voltar.

Aula 2.2 – Electrum (BTC) com Tor – sem restore height, mas com tipo de carteira

Objetivo: Instalar Electrum, configurar proxy Tor, criar carteira Segwit, entender que a restauração só precisa da seed + o tipo de carteira.

Terminal – iniciar Electrum com Tor (no Tails já vem instalado): 

# No Tails, o Electrum está em Applications → Internet → Electrum
# Para garantir o proxy Tor, abra o Electrum e vá em:
# Tools → Network → Proxy → Use Tor proxy (localhost:9050, SOCKS5)

Criação da carteira: 

# Se não estiver no Tails, instale:
sudo apt install electrum -y   # Debian/Ubuntu
# Ou baixe o AppImage

Na interface:

  • File → New/Restore → dar nome.
  • Standard wallet → Create a new seed.
  • Escolha Segwit (endereços começam com bc1). Anote a seed (12 palavras).
  • IMPORTANTE: Anote junto com a seed: Tipo: Segwit. Se você errar o tipo na restauração, o saldo vai aparecer zerado.
  • Defina uma senha para o arquivo da carteira.

Restauração correta (simulação): 

# Delete o arquivo da carteira (feche o Electrum antes)
rm ~/.electrum/wallets/minha_carteira
# Reabra o Electrum → Restore → digitar a seed → escolher Segwit → saldo reaparece

Teste de recebimento: 

# Dentro do Electrum, vá em Receive → criar um novo endereço (começa com bc1)
# Use esse endereço para receber uma pequena quantia.

Modelo de ameaça: O proxy Tor esconde seu IP. O tipo de carteira correto evita sustos na restauração.

Desafio: Crie uma carteira Segwit, anote a seed e o tipo. Delete e restaure escolhendo o tipo correto.

# No Cake Wallet: Settings → Node → Add custom node → digite o IP do seu nó.

Módulo 3 – O bunker: Tails + Persistência + Backup

Aula 3.1 – Tails do zero: instalação e primeira inicialização

Objetivo: Gravar o Tails em um pendrive A, criar a persistência LUKS, configurar senha forte.

Terminal – verificação da imagem (no Linux hospedeiro, antes de gravar): 

# Baixe a imagem .img do site oficial tails.net
# Importe a chave GPG do Tails
gpg --auto-key-locate nodefault,wkd --locate-keys tails@boum.org

# Verifique a assinatura
gpg --verify tails-amd64-*.img.sig tails-amd64-*.img
# Deve aparecer "Good signature"

Gravação com balenaEtcher (interface gráfica):

  • Abra o balenaEtcher.
  • Selecione a imagem .img do Tails.
  • Selecione o pendrive A (cuidado para não escolher o HD).
  • Clique em Flash.

Primeiro boot e criação da persistência:

Ao iniciar pelo pendrive A:

  • Na tela de boas-vindas, escolha Start Tails.
  • Após carregar, vá em Applications → Tails → Configure persistent storage.
  • Escolha o pendrive A, defina uma senha forte (12+ caracteres).
  • Ative os recursos: Personal Data, Network Connections, GnuPG (opcional).
  • Clique em Save, reinicie.

No próximo boot, desbloqueie a persistência marcando a opção Encrypted Persistent Storage e digitando a senha.

Verificação no terminal: 

# Após desbloquear, monte a persistência (já deve estar montada)
ls /live/persistence/TailsData_unlocked/
# Deve mostrar pastas como dotfiles, gnupg, etc.

Modelo de ameaça: LUKS protege contra roubo do pendrive. A senha forte evita força bruta.

Desafio: Crie seu pendrive Tails, configure persistência, reinicie e desbloqueie. Rode ls /live/persistence/TailsData_unlocked/ e veja as pastas.

Aula 3.3 – rsync: o copiar e colar que não quebra (com proteção)

Problema: Copiar com o mouse ou cp quebra permissões, links e arquivos esparsos.
Analogia: rsync é um robô engenheiro; copiar/colar é um estagiário desastrado.

Comando completo (apenas após verificação): 

# PASSO 1: Verifique se a origem (persistência desbloqueada) NÃO está vazia
ls -la /live/persistence/TailsData_unlocked/
# Deve mostrar várias pastas (Feather, Electrum, etc.)

# PASSO 2: Execute um dry-run (simulação) para ver o que seria copiado/apagado
sudo rsync -PaSHAXv --dry-run --del /live/persistence/TailsData_unlocked/ /media/amnesia/TailsData/

# PASSO 3: Se o dry-run mostrou apenas os arquivos esperados, remova --dry-run e execute
sudo rsync -PaSHAXv --del /live/persistence/TailsData_unlocked/ /media/amnesia/TailsData/

Explicação das flags:

  • -P → progresso e partial (pode retomar)
  • -a → archive (preserva permissões, donos, timestamps)
  • -S → trata arquivos esparsos (essencial para banco de dados do Haveno)
  • -H → preserva hard links
  • -A → preserva ACLs
  • -X → preserva atributos estendidos
  • -v → verbose (mostra cada arquivo)
  • --del → deleta no destino o que não existe na origem (espelhamento)

Demonstração de falha proposital (NUNCA faça isso com dados reais): 

# Se a origem estiver VAZIA (persistência não desbloqueada), o --del apagaria o backup
# Por isso o dry-run é obrigatório.
# Exemplo de como NÃO fazer:
sudo rsync -PaSHAXv --del /live/persistence/TailsData_unlocked/ /media/amnesia/TailsData/
# Se /live/persistence/TailsData_unlocked/ estiver vazio, o rsync apagará tudo no destino.

Modelo de ameaça: Protege contra falha de hardware (pen drive A queimar). Mantém o pen drive B como clone exato.

Desafio: Execute o rsync com --dry-run primeiro. Confirme que a lista de arquivos está correta. Depois rode o comando real.

Aula 3.4 – Tails Cloner: atualizando o sistema do pen drive B

Objetivo: Atualizar a partição de sistema do pen drive B sem perder os dados da persistência.

Regra de ouro operacional: Primeiro atualize o sistema (Cloner), depois sincronize os dados (rsync).

Terminal – clonagem segura (mesma versão principal):

  1. Dê boot no pen drive A (já atualizado via internet ou ISO nova).
  2. Conecte o pen drive B (alvo).
  3. Não desbloqueie a persistência do B (se o Tails perguntar, clique em Cancelar).
  4. Abra Applications → Tails → Tails Cloner.
  5. Selecione o dispositivo alvo (pen drive B).
  6. Leia a mensagem: “An existing Persistent Storage was found. It will be preserved.” – só prossiga se aparecer isso.
  7. Clique em Install.
  8. Aguarde.

Após o clone, sincronize os dados com rsync (aula 3.3).

Saltos grandes de versão (exemplo hipotético – verifique as versões reais em tails.net):

  • NÃO use o Tails Cloner diretamente de uma versão muito antiga para uma nova.
  • Procedimento seguro:
    1. Instale o Tails novo em um pendrive C (limpo).
    2. Boot no C, crie uma persistência nova (vazia).
    3. Conecte o pendrive antigo (A) e desbloqueie sua persistência.
    4. Copie apenas as pastas de dados (Feather, Electrum, Haveno) via rsync do A para o C.
    5. Teste o C. Se tudo funcionar, ele se torna seu novo pen drive principal.

Modelo de ameaça: Mantém o sistema operacional atualizado (correções de segurança do Tor e kernel) sem perder seus dados.

Desafio: Se você tem duas versões do Tails, pratique a clonagem segura. Se não, simule com um segundo pendrive vazio.

Módulo 4 – Avançado: soberania total e resiliência extrema

Aula 4.1 – VeraCrypt dentro do Persistent – negação plausível (fluxo correto para Tails)

Ponto crítico: O Tails não cria volumes VeraCrypt – ele apenas abre volumes já existentes (via Applications → Tails → Unlock VeraCrypt Volumes).
Portanto, você deve criar o contêiner (volume normal + oculto) em uma máquina offline confiável (ex: um Linux em outro computador, ou uma VM sem rede).

Criação do volume fora do Tails (ex: num Ubuntu offline): 

# Instale o VeraCrypt na máquina offline
sudo apt install veracrypt -y

# Crie um arquivo contêiner de 500 MB
dd if=/dev/zero of=isca.hc bs=1M count=500

# Crie o volume normal (isca) – use a senha A
veracrypt --text --create isca.hc
# Escolha: 1 (Normal volume), AES, SHA-512, tamanho 500 MB, senha A, FAT32

# Agora crie o volume oculto DENTRO do mesmo arquivo
veracrypt --text --create isca.hc
# Escolha: 2 (Hidden volume), senha A (para o volume normal), depois senha B (para o oculto)
# Tamanho do oculto: até ~490 MB (deixe um pouco de espaço livre)

Copie o arquivo isca.hc para o seu Persistent Storage do Tails (via pendrive criptografado ou rede local segura).

Dentro do Tails (para montar): 

# Primeiro, desbloqueie a persistência LUKS normal
# Depois, abra o Unlock VeraCrypt Volumes (menu Applications → Tails)
# Ou pela linha de comando:
veracrypt --text --mount /live/persistence/TailsData_unlocked/VeraCrypt/isca.hc /mnt/isca
# Digite a senha A (volume isca) ou senha B (volume oculto)

Onde guardar a seed principal?
A seed principal (o tesouro) vai em papel/metal, nunca dentro do volume oculto. Dentro do volume oculto coloque backups secundários (ex: cópia das chaves públicas, histórico de trades, seeds de carteiras de teste).

Modelo de ameaça: Coerção física – você entrega a senha A, o agressor vê arquivos fakes e vai embora.

Desafio: Crie um volume VeraCrypt com volume oculto em uma máquina offline. Copie para seu Tails e pratique montar o volume isca e o oculto.

Aula 4.2 – Seu próprio nó Monero (full ou podado) com hardware velho

Tamanhos padronizados (2025):

  • Full node: 220‑230 GB (+30 GB/ano)
  • Pruned node: ~95 GB (+10 GB/ano)

Instalação do monerod no Linux (Ubuntu/Debian): 

# Baixar o binário oficial
wget https://downloads.getmonero.org/cli/linux64
tar -xjvf linux64
cd monero-x86_64-linux-gnu-v*
sudo cp monerod monero-wallet-cli monero-wallet-rpc /usr/local/bin/

# Criar diretório da blockchain (em SSD, não HDD)
sudo mkdir -p /var/lib/monero
sudo chown -R $USER:$USER /var/lib/monero

# Criar arquivo de configuração ~/.bitmonero/bitmonero.conf
cat > ~/.bitmonero/bitmonero.conf << EOF
data-dir=/var/lib/monero
prune-blockchain=1            # descomente para nó podado (remova o #)
# limit-rate-up=1024
# limit-rate-down=2048
rpc-bind-ip=127.0.0.1        # somente local. Para rede interna use 192.168.x.x
rpc-bind-port=18081
confirm-external-bind=0      # NUNCA abra 18081 para a internet
EOF

# Iniciar o daemon
monerod --config-file ~/.bitmonero/bitmonero.conf

Acompanhe a sincronização (pode levar dias): 

monerod status
# Exemplo de saída: Height: 3200000/3200000 (100%) on mainnet, synced

Firewall (libere apenas P2P, mantenha RPC fechada): 

sudo ufw allow 18080/tcp comment 'Monero P2P'
sudo ufw allow from 192.168.1.0/24 to any port 18081/tcp comment 'Monero RPC local (opcional)'
sudo ufw enable

Conecte sua Feather Wallet ao nó local:

  • Abra Feather → Settings → Node → Custom node
  • Endereço: 127.0.0.1:18081 (se for no mesmo PC) ou 192.168.1.100:18081 (rede local)

Modelo de ameaça: Soberania total – você não depende de nós de terceiros e valida suas próprias transações.

Desafio: Monte um nó podado em um PC velho ou VM. Deixe sincronizar e conecte sua Feather a ele.

Aula 4.3 – Whonix para traders frequentes (com pontos cegos)

Arquitetura: Gateway (Tor) + Workstation (aplicações). O tráfego da Workstation sempre passa pelo Gateway.

Instalação (no VirtualBox): 

# No sistema hospedeiro (ex: Ubuntu), baixe as imagens do site whonix.org
# Importe as duas VMs: Whonix-Gateway e Whonix-Workstation
# Inicie primeiro o Gateway, depois a Workstation

Dentro da Workstation (terminal): 

# Instale o Feather
wget https://featherwallet.org/files/releases/feather-2.6.4.AppImage
chmod +x feather-*.AppImage
./feather-*.AppImage

# Instale o Haveno (similar)

Pontos cegos (obrigatório ensinar):

  1. Host infectado: Se o sistema operacional principal (hospedeiro) tiver um keylogger, o atacante captura tudo que você digita na VM. O Whonix não protege contra isso.
  2. Latência e timeouts: O Haveno exige conexões P2P estáveis. Rodá-lo sobre Tor + duas VMs pode causar quedas de conexão no meio de um trade.

Vantagens: Isolamento de rede robusto – mesmo que a Workstation seja comprometida, o IP real não vaza.

Backup: Faça snapshots da VM no VirtualBox (VM → Take Snapshot).

Modelo de ameaça: Protege contra vazamento de IP através de aplicativos que tentam se conectar diretamente.

Desafio: Instale o Whonix, dentro da Workstation instale Feather e conecte a um nó remoto. Teste com uma transação pequena.

Aula 4.4 – Erros humanos que quebram toda a criptografia

Lista dos erros mais comuns (leia em voz alta durante a aula):

  1. Senha fraca no LUKS/carteira → use openssl rand -base64 20 para gerar senhas fortes.
  2. Foto da seed no celular → a nuvem vaza. Solução: papel ou metal.
  3. Falar sobre seu setup para pessoas erradas → opacidade total.
  4. Não testar o backup → faça boot no pendrive B a cada 30 dias.
  5. Login em contas pessoais dentro do Tails → quebra o anonimato.
  6. Não ter backup físico da seed em metal → papel queima/molha.
  7. Usar o mesmo pendrive por anos sem substituir → ciclos de escrita matam NAND.

Checklist de segurança comportamental (para imprimir e colar): 

# Gere um arquivo de checklist na sua área de trabalho (Linux)
cat > ~/Desktop/checklist_seguranca.txt << EOF
[ ] Seed principal em metal + papel (dois locais diferentes)
[ ] Nenhuma foto da seed no celular ou nuvem
[ ] Senha LUKS tem 12+ caracteres (não é data de nascimento)
[ ] Backup rsync testado nos últimos 30 dias
[ ] Ninguém (além de mim) sabe que tenho cripto
[ ] Pendrive Tails tem menos de 18 meses de uso
EOF

Modelo de ameaça: Protege contra você mesmo – o elo mais fraco da cadeia.

Desafio final: Imprima a checklist. Marque os itens que você já cumpre. Corrija os que estão faltando. Poste: “Minha segurança comportamental está em dia.”

🎁 Aula bônus – Simulação de falha do pendrive A

Objetivo: Testar se o pen drive B realmente funciona como backup.

Terminal – roteiro da simulação: 

# 1. Antes de começar, verifique se o B está atualizado (rsync e Cloner já rodados)

# 2. Desligue o computador e retire o pendrive A. Guarde-o.

# 3. Ligue o computador apenas com o pendrive B. Dê boot.

# 4. Desbloqueie a persistência do B (digite sua senha).

# 5. Abra o terminal e verifique a integridade:
ls /live/persistence/TailsData_unlocked/
# Deve mostrar as pastas Feather, Electrum, etc.

# 6. Abra a Feather:
/live/persistence/TailsData_unlocked/Feather/feather-*.AppImage
# Confirme se o saldo está correto (deve ser o mesmo do A após o último rsync)

# 7. Se tudo funcionar, seu backup está aprovado. Se falhar, refaça o rsync e o Cloner.

Script de verificação automática (opcional): 

#!/bin/bash
echo "=== Teste de integridade do backup ==="
[ -f /live/persistence/TailsData_unlocked/Feather/feather-*.AppImage ] && echo "✅ Feather presente" || echo "❌ Feather ausente"
[ -d /live/persistence/TailsData_unlocked/Electrum/data/.electrum ] && echo "✅ Electrum configurado" || echo "❌ Electrum sem configuração"
echo "=== Fim do teste ==="

Desafio bônus: Execute a simulação de falha. Grave um pequeno vídeo (pode ser privado) de você bootando no B e abrindo a Feather com o saldo correto.
